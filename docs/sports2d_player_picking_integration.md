# Sports2D Player Picking Integration

This document outlines the architectural and codebase changes introduced to support the native Sports2D player picking integration. By transitioning from a single-step upload-and-analyze model to a two-step process utilizing a "picker session", the system now enables user interaction between uploading the video and running the final analysis.

## 1. New "Picker Session" Endpoints

Instead of immediately jumping into the analysis, the system now has dedicated endpoints to manage a "picker session," which processes initial video data to allow the frontend to run Sports2D's player picking first.

The following endpoints were created:

* **`POST /picker/sessions`**
  Uploads a video and initializes a picker session. Saves the file to a `mitus_picker_uploads` temporary directory and begins parsing initial video metadata.

* **`GET /picker/sessions/{session_id}`**
  Polls the current status of the player picking operation, allowing you to show loading states and wait for coordinate outputs.

* **`POST /picker/sessions/{session_id}/finalize`**
  Signals that the player picking operation by the user is complete and safely finalizes the session's outputs so it is ready for analysis.

* **`POST /picker/sessions/{session_id}/cancel`**
  Halts the picker process to free up server resources if the user backs out.

* **`DELETE /picker/sessions/{session_id}`**
  Cleans up temporary files generated by the picker when no longer needed.

## 2. Overhaul of the `/analyze` Endpoint

The main `/analyze` route was significantly modified to support the new two-step interaction while maintaining backward compatibility with the old direct upload method.

* **Form Data Transition:** 
  Because file uploads are now mixed with other parameters in one `multipart/form-data` request, all parameters (`player_id`, `yolo_size`, `session_tags`, etc.) were updated to utilize FastAPI's `Form(...)` wrapper instead of query parameters.
  
* **Optional File Upload Mechanism:**
  The `file` parameter is now optional (`Optional[UploadFile]`). This is because the user might have already uploaded the video during the picker session step.

* **Picker Reference parameter (`picker_session_id`):** 
  A new optional `picker_session_id` parameter was added to the endpoint.
  * **When provided:** The backend pulls the previously uploaded video from the picker session and verifies that the picker is finalized and ready. 
  * **When omitted:** The system falls back to the legacy behavior, securely processing the provided file in a single all-in-one step.

* **Bypassing Redundant Extraction:** 
  Instead of running the tracker pipeline from scratch, the API now resolves the output directory containing the Sports2D extraction (`s2d_source_dir`). This cached work is forwarded directly into the background job, drastically saving redundant GPU and calculation time.

## 3. Updates to the Background Worker

The signature for the `run_full_analysis_job` background task was updated to coordinate this new flow smoothly:

* New arguments, `s2d_source_dir` and `pick`, were added to instruct the underlying worker thread to pull tracking and inference data from the previously executed picker session state.
* Robust logging has been wrapped in fallback mechanisms (e.g., `try: ... except: pass`). This ensures that trivial database errors—such as network hiccups while logging real-time status to Supabase—do not crash or interrupt the heavy background process.

## Summary for Frontend Implementations

To consume this new flow on a web client, execute the following steps:

1. Send a request to `POST /picker/sessions` to upload the video and start the initial tracker parsing.
2. Periodically poll `GET /picker/sessions/{session_id}` to provide user feedback when it completes.
3. Present your UI (web app interface) for picking the player based on the session's output targets.
4. Call `POST /picker/sessions/{session_id}/finalize`.
5. Finally, hit `POST /analyze`, providing the `picker_session_id`. This will bypass the second video upload and rapidly trigger the core analytics based on the user's specific bounding box selection.
